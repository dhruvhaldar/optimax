## 2024-05-23 - [Information Leakage via Exception Handling]
**Vulnerability:** The backend API endpoints were catching all exceptions with `try...except Exception as e:` and returning the raw error message to the client (`detail=str(e)`). This exposed internal implementation details, such as `scipy` or `pulp` error messages, which could potentially leak system paths or sensitive data, and provided attackers with information about the backend technology stack.
**Learning:** Developers often use broad exception handlers to quickly debug issues during development, but forget to sanitize messages for production. In scientific computing APIs (like this one using `scipy`), `ValueError` often maps to user input errors (e.g., matrix dimension mismatch), while other exceptions are typically system errors.
**Prevention:** Use FastAPI's `@app.exception_handler` to globally handle exceptions. Map known safe exceptions (like `ValueError` for bad input) to generic 400 responses with sanitized messages. Catch all other exceptions as 500 "Internal Server Error" and log the full traceback server-side only. Do not rely on `try...except` blocks in individual route handlers unless specific recovery logic is needed.

## 2024-10-24 - [Unbounded Input Size (DoS)]
**Vulnerability:** Pydantic models for mathematical solvers (`List[float]`, `List[List[float]]`) accepted lists of arbitrary size. This could allow an attacker to send massive matrices, causing memory exhaustion (OOM) or excessive CPU usage in the backend solvers (`scipy`, `pulp`), leading to a Denial of Service.
**Learning:** Type hints like `List[float]` validate the *type* of elements but not the *quantity*. For computational endpoints, explicit bounds are critical to protect resources.
**Prevention:** Use `Annotated[List[T], Field(max_length=N)]` (Pydantic v2) or `conlist` (Pydantic v1) to enforce strict size limits on all list inputs.

## 2024-10-25 - [Process Exhaustion via Subprocess Spawning]
**Vulnerability:** The use of `pulp` with its default CBC solver spawned a new subprocess for every subproblem solved in iterative algorithms (like Lagrangian relaxation). In high-concurrency scenarios or with complex problems requiring many iterations, this could lead to process table exhaustion (DoS) and increased attack surface via shell execution.
**Learning:** While `pulp` is convenient for modeling, its default behavior of shelling out to solvers is risky in production environments, especially serverless or containerized ones where process creation is expensive or limited. `scipy.optimize.milp` (available in newer SciPy versions) offers a high-performance, in-process alternative using HiGHS.
**Prevention:** Prefer in-process solvers like `scipy.optimize.milp` or `linprog` over wrappers that spawn subprocesses (`pulp`, `pyomo` with external solvers) for high-frequency or user-driven computations.

## 2026-03-06 - [Log Flooding/DoS via Missing Dimensionality Validation]
**Vulnerability:** While Pydantic models restricted the total length of lists (`BoundedFloatList`), they lacked exact length validation for nested arrays representing specific mathematical entities (e.g., a `demand` tuple must have exactly `[width, quantity]`). Sending malformed arrays with missing dimensions bypassed initial validation but caused unhandled `IndexError` exceptions deep within the SciPy solver functions (e.g., `yields[2]`). While caught by a global exception handler and returning a generic 500 error, this generated massive stack traces in the server logs for each request, allowing an attacker to easily cause log flooding and potentially CPU exhaustion (DoS).
**Learning:** Type hints and generic length limits (e.g., `List[float]`) are insufficient for mathematical endpoints where specific array dimensions are algorithmically required.
**Prevention:** Use Pydantic's `min_length` and `max_length` together inside `Annotated[..., Field()]` to strictly define the exact shape of fixed-size nested arrays (e.g., `YieldTuple = Annotated[List[SafeFloat], Field(min_length=3, max_length=3)]`). This ensures validation fails instantly at the ingress point, returning a standard 422 Unprocessable Entity without triggering any backend compute or exception stack traces.

## 2024-10-26 - [Out-Of-Memory via Unbounded Knapsack DP Array Size]
**Vulnerability:** The Column Generation solver uses a custom 1D dynamic programming (DP) array to solve the unbounded knapsack subproblem for maximum performance. However, the size of this array (`dp = np.zeros(W + 1, dtype=float)`) scales directly with the user-provided `roll_length` (W). An attacker could submit an extremely large `roll_length` (e.g., 1e9), causing the server to attempt allocating massive contiguous memory chunks, leading to immediate Out-Of-Memory (OOM) crashes and DoS.
**Learning:** While replacing standard generalized solvers (like Scipy MILP) with custom algorithmic implementations (like DP) provides massive performance benefits, it often exposes raw computational parameters directly to user input, bypassing generic safety constraints.
**Prevention:** Always implement explicit, hardcoded size limits or guard clauses on algorithmic parameters that directly dictate memory allocation sizes or loop bounds, preventing users from triggering disproportionate resource consumption.

## 2026-03-07 - [Server Error/Information Leakage via Empty List Inputs]
**Vulnerability:** Endpoints handling arrays of data (`c`, `A_ub`, `b_ub`, `demands`, `costs`, `weights`) lacked validation for minimum length limits. Supplying an empty list bypassed length validations but triggered `ValueError` or `IndexError` inside backend solvers (like SciPy's `linprog` or `milp`). This resulted in an unhandled exception cascading to the global exception handler, sending a 500 error, and logging the internal stack trace, allowing attackers to leak framework-specific trace details and waste server resources.
**Learning:** Type limits (e.g. `List[float]`) and maximum limits (`max_length=N`) protect against OOM and log flooding from massive arrays, but zero-length bounds (`min_length=1`) are equally important when the arrays represent mathematical entities passed straight to external solvers that expect non-empty arguments.
**Prevention:** Use Pydantic's `min_length=1` parameter in `Field()` inside `Annotated[..., Field()]` definitions (e.g., `BoundedFloatList = Annotated[List[SafeFloat], Field(min_length=1, max_length=MAX_VARS)]`) to strictly enforce that required input matrices are not empty. This blocks malformed inputs at the API gateway via a 422 Unprocessable Entity response before processing begins.

## 2024-04-03 - [Strict Probability Types and Non-Cacheable Results]
**Vulnerability:** The `Scenario` probability fields were validated merely as `SafeFloat` (-1e20 to 1e20) instead of being strictly bounded to a [0, 1] range. Additionally, solver endpoint responses containing potentially sensitive organizational optimization data were missing the `Cache-Control` header, allowing local or proxy caching.
**Learning:** Using overly permissive float validations can lead to logic flaws or unpredictable execution inside scientific solvers like Scipy if constraints like probabilities are not strictly enforced natively at the API boundary using Pydantic.
**Prevention:** Always construct domain-specific boundary constraints via Pydantic (e.g., `Field(ge=0.0, le=1.0)`) instead of relying on generic safety constraints. In addition, universally apply `Cache-Control: no-store, no-cache, must-revalidate, max-age=0` to API responses containing internal computational data to defend against local or intermediate caching vulnerabilities.
